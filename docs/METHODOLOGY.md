# Helm 自动化开发方法论

> 适用对象:用 `/loop` + feature-room skills + VibeHub 驱动的**真·无人值守**自动开发。
> 这份文档同时是**给人看的完整版**;其中的"升级边界"与"自决/留痕清单"会被精简后写入 VibeHub 约束(`add_constraint`),好让每一轮 loop 通过 `get_team_conventions` 自动读到。
>
> 核心信念:让全自动安全的,不是"原则",而是 **错了也便宜(可回滚)、错了也看得见(留痕)、撞墙会停(熔断)**。原则只决定"每一步怎么想"。
>
> **本档位的契约:只有「难撤 / 定方向的产品决策」需要人;技术决策(含数据模型、对外契约、安全边界、核心选型)+ 可逆的产品小决定(文案、默认排序…)都 loop 自决,按可逆性分级详细留痕,Room 做完全自动合 main。** 安全完全靠验证门 + 熔断 + 分级留痕兜底。
>
> ⚠️ 下面都是**默认倾向,不是死规矩**。拿不准时从宽,优先把活往前推,别为了走流程牺牲进度。阈值/清单都是参考,可以凭判断灵活处理。

## 一、六条原则(决策链,按顺序执行)

遇到任何 OQ(Open Question)/ 不确定点,按以下顺序判断:

### 1. 升级边界(先判该不该自己答)
> 档位 = 真·无人值守。**两道判据叠加**:先看**领域**(技术 vs 产品);产品再看**可逆性**。

- **技术决策** → **loop 自己决,不停**。含数据模型 / schema、对外契约 / MCP 签名、**安全边界**、核心选型、跨 Room 抽象,以及一切内部实现。按可逆性**分级留痕**(第 4 条),但不问人。
- **产品 / 业务决策** → 再按可逆性分流:
  - **低风险、可逆**(纯文案 / 措辞、默认排序、默认筛选、空状态文案、列表默认顺序、占位符、默认标签页等——改回来只是一次**无数据迁移的小 commit**)→ **loop 自己定 + `record_decision`**,不打扰你。
  - **难撤 / 定方向**(改用户可感知的数据语义、不可逆流程、用户会形成依赖的产品方向、跨 Room 产品契约、功能取舍 / 优先级 / 范围)→ **停下问人**:`add_question` + 留显式 TODO;**别写隐式假设了某个答案的代码**;阻塞本 milestone 则熔断停(第三节)。
- **拿不准某个产品决策可不可逆 → 当"难撤"处理、上报**(保守默认,见第 3 条:安全 > 速度)。技术决策的可逆性只决定**留痕详细度**(第 4 条),不决定问不问人。

### 2. 有据可依(决策必须有来源)
- 决策只能来自:PRD(`docs/prd/`)/ VibeHub spec / 已有代码 / 已记录的约束。
- **找不到依据 ≠ 猜**。技术决策找不到依据时,选**最安全、最易回滚**的解法(第 3 条),并把"无明确依据、取了保守解"写进 `record_decision`。
- 只有当无依据的问题**本质是难撤/定方向的产品决策**时才 defer 问人;技术 + 可逆产品小决定就自己保守定 + 留痕。

### 3. 取舍排序(用于一切自决的技术决策)
- **安全 > 正确/完整 > 速度**。
- 同一问题有"安全但费事"和"省事但有风险"两种解法时,选前者。
- 高风险技术决策(见第二节)在没有明确依据时,这条是默认裁决依据。

### 4. 留痕(自动决策按可逆性分级记录)
> 全自动下,**记录就是人事后唯一的介入抓手**,所以从详不从简。可逆性(改起来贵 / 被其它模块依赖 / commit 后难撤,三者任一 = 高风险)决定记录的详细度:

- **每个**自决技术决策 → `record_decision`,至少写**理由 + 依据来源**。
- **高风险技术决策**(数据模型 / schema、对外契约 / MCP 签名、**安全边界**、核心选型、跨 Room 抽象):`record_decision` **写满**——理由 + 依据 + **影响范围 + 考虑过的备选 + 为什么不选**;涉及硬边界再 `add_constraint`;并在本轮 report 里**显著置顶标注**,确保 dashboard 上一眼可审、可回滚。
- **低风险技术决策**(命名、内部结构、helper、测试组织、日志/文案):`record_decision` 一句话理由即可。
- 不留痕的后果:下一轮 loop 会重问、甚至答得不一致 → 决策漂移;且人失去了对全自动的唯一审计抓手。

### 5. 验证后才提交(commit-sync 前的硬门)
- 提交前必须过:**测试 + 构建/类型检查**(项目早期至少要能 import / 启动成功)。
- 没有验证手段的代码 = 不可全自动。补齐验证门是上"真·无人值守"的前置条件。
- 验证失败 → 不提交,进入熔断判断(见第三节)。

### 6. 有限爆炸半径(约束影响范围)
- **一个 Room = 一条分支**(如 `feat/platform-shell`)。绝不在 `main` 上自动开发。
- 在该分支上把这个 Room 的 milestone 逐个做完;**单 milestone 单 commit**,便于整段回滚。
- Room 收尾 → Room 级优化扫描(前端 + 后端)→ 全量验证门 → 产出复盘(`room-status`)→ **全绿即全自动合回 `main`**(开 PR 留 GitHub 审计 + 自动 squash-merge);**任一门未过则停下等人**,不硬合。
- **下一个 Room 从最新的 `main` 切新分支**,这样它能拿到前序 Room 已合入的成果(处理 Room 间依赖,如大家都依赖 `platform-shell` 地基)。
- 因此 **Room 之间是串行**的:做完一个、合一个、再切下一个,避免并行分支互相依赖时冲突。
- 爆炸半径的安全此时全靠**验证门 + 熔断**兜底(不再有人工合并卡点)——所以验证门必须是真的、熔断必须真会停。

## 二、Helm 的升级边界清单(问人 vs 自决)

> 把第一节原则落到本项目的清单,当参考用。**只有「难撤的产品决策」停下问人;技术决策 + 可逆的产品小决定都 loop 自决 + 留痕**。

### 需要问人 —— 难撤 / 定方向的产品决策(loop 不该自己定)

- 改**用户可感知的数据语义**、不可逆流程、用户会形成依赖的产品方向。
- **功能取舍 / 优先级 / 范围**:做不做、先做哪个、做到什么程度。
- **跨 Room 产品契约**(一个 Room 的产品行为被别的 Room 依赖)。
- 命中这些 → `add_question` + 建 issue + 代码留显式 TODO;**阻塞本 milestone 则熔断停**。

### loop 自决 + 留痕 —— 可逆的产品小决定

> 判据:改回来只要**一次无数据迁移的小 commit**、用户不会因此形成难撤的依赖。拿不准 → 当"难撤"上报。

- 纯文案 / 措辞 / 报错提示 / 提示文案、按钮与标签文字。
- 默认排序、默认筛选、默认标签页、列表默认顺序、空状态文案、占位符。
- 非结构性的 UI 默认值(可一键改回的那种)。
- 这些 → 自己定 + `record_decision`(一句话理由即可),写进 report,你可在 dashboard 事后否决。

### loop 自决 + 留痕 —— 一切技术决策

> 按第一节第 4 条**分级记录**。下面前一组是**高风险**(`record_decision` 写满 + 可能 `add_constraint` + report 置顶标注),后一组低风险(一句话理由即可)。

- **高风险(满格留痕,report 置顶)**:
  - **数据模型 / 本地存储 schema**(会导致迁移的结构)。
  - **对外契约**:暴露给 Claude Code 的 MCP 工具签名与语义(`agent-orchestration`);跨 Room 契约(`cockpit` 如何观测 agent 产出、`memory-rag-skills` 的 MCP 表面)。
  - **安全边界**:凭据/密钥的存储位置与加密方式;哪些数据发往外部 provider(`chat-multimodel` / `deep-research`);终端 agent 的沙箱与权限边界。
  - **核心选型**:后端框架、桌面外壳、关键第三方依赖、打包方案的最终确定。
  - **跨 Room 共享的基础抽象**:多个 feature 会依赖的公共模块接口。
- **低风险(一句话理由)**:Room 内命名、文件/目录组织、helper 实现、测试组织、日志/文案/注释、内部重构。

> ⚠️ **安全边界本来"错了很贵"**,现在交给 loop 自决,是以「**详细留痕 + 熔断**」换「无人值守」的明确取舍。
> 因此这类决策**必须** `add_constraint` + 在 report 置顶,确保 dashboard 上一眼可审、可回滚——
> 这是把高风险技术决策放给 loop 的前提,不是可选项。

## 三、熔断条件(任一触发 → 停止 loop,等人)

全自动的安全性主要来自"会停下来"。下面阈值是**参考值,不用死扣**;命中其一,大致就该停下来产出状态摘要、等我介入:

1. **测试/构建连续失败 2 次**(同一 milestone)。
2. **难撤产品 OQ 阻塞**:出现 ≥1 个属于「难撤 / 定方向」产品决策、loop 不该自己定的 OQ,且它阻塞了后续工作。(技术决策、可逆的产品小决定都不算——loop 自决并留痕即可。)
3. **打转**:同一文件在一轮内被反复修改 ≥3 次仍未通过验证。
4. **defer 堆积**:单个 Room 累计未解决 issue/question ≥ 5 条。
5. **范围漂移 / 越界**:实际改动超出当前 milestone 声明的范围(commit-sync 一致性检查不通过),或触及难撤产品决策而本 milestone 未授权。
6. **依据缺失(产品侧)**:连续 ≥2 个**难撤产品** OQ 因"找不到依据"而无法推进。(技术、可逆产品小决定无依据时取保守解 + 留痕,不熔断。)
7. **合并门未过**:Room 收尾时全量验证 / 优化扫描 / CI 未全绿——**不合 main**,停下等人。

熔断时必须输出:停在哪个 Room/milestone、触发了哪条、已 defer 的 issue 列表、建议的下一步。

## 四、自动化成熟度分档

| 档位 | 前置条件 | 人的角色 |
|------|----------|----------|
| **监督式自动** | 六条原则 + 升级边界清单 + 熔断已配置 | 每个 Room 跑完看 `room-status` 复盘、人工合 PR |
| **真·无人值守(当前档位)** | 额外需要:验证门(测试/CI)就绪、熔断真的会触发、dev 分支 + 单 milestone 单 commit、技术决策分级留痕 | **仅业务/产品 OQ 与熔断节点介入**;事后在 dashboard 审计高风险决策 |

**当前状态**:`platform-shell` 起验证门(`pytest` / `vitest` / 构建)已就绪、熔断已配置 → **切换到真·无人值守**:loop 自动 plan / 决策 / 优化 / 合并,只有业务/产品 OQ 与熔断节点需要人。

**前提(逐 Room 检查)**:每个 Room 的关键路径必须有能 **headless 断言**的验证门;若某 Room 验证手段不足(只能人工目视),它**不具备无人值守前置条件** → 该 Room 退回监督式(做完停下、等人 review 合 PR),别全自动合 main。

## 五、Milestone 拆分原则(plan-milestones 阶段适用)

> 目标:**能在当前 milestone 做完的,就当场做完,不要顺手丢一个 "留到 phase 5" 的 TODO。** 拖延式拆分会让代码长期处于半成品状态、stub 满天飞,最后没人记得回去补。

1. **就近做完(close it out)**:一个功能如果在当前 milestone 的能力范围内就能完整实现,就完整实现掉,不要拆成"先搭壳、后面再填"。
2. **垂直切片优于水平分层**:每个 milestone 交付一个**端到端可验证**的薄切片(从数据到 UI 打通一条),而不是"这个 milestone 只搭后端、UI 等以后"。完成后这块应当是真能跑、能验证的,而非一堆待接线的桩。
3. **禁止隐性拖延**:不允许用 "TODO: phase N 再做" 来回避当前就能完成的工作。真正需要推迟的,只有两类——
   - **难撤/定方向的产品 OQ**(需要人拍板,见第二节)→ 走 defer + 建 issue,这是合法推迟。(技术决策、可逆产品小决定都不算——自己定 + 留痕即可,不推迟。)
   - **明确超出本 Room 范围**的工作 → 记成另一个 Room/feature 的 issue。
   除此之外,"能做但懒得现在做"不是合法的推迟理由。
4. **拆分粒度**:milestone 要小到"单 milestone 单 commit、可整段回滚"(原则 6),但不要小到把一个内聚的功能硬切成"搭壳 / 填肉"两半。**以"完成一个可验证的能力"为单位,而不是以"碰一层代码"为单位。**
5. **完成定义(DoD)**:一个 milestone 算完成 = 该切片功能可用 + 通过验证门(原则 5)+ 没有为了赶进度留下的隐式假设/半接线 stub。

冲突时的优先级:**完整地做完一个小切片 > 铺开很多个半成品**。

## 六、单轮 loop 的标准动作

```
1. pull context:get_feature_context / query_why / get_team_conventions
2. 取下一个 milestone(无 → Room 收尾:限额 drain 技术 backlog → 优化扫描[前端+后端] → 全量验证 → room-status → 自动合 main)
3. 开发;遇 OQ → 走第一节"升级边界"决策链
   - 技术决策(含原单向门):自己决 + 按可逆性分级 record_decision(高风险再 add_constraint + report 置顶)
   - 产品·可逆小决定(文案/默认排序…):自己定 + record_decision(一句话)
   - 产品·难撤/定方向:add_question + 建 issue + 留显式 TODO(阻塞则熔断停)
4. review(/code-review)
5. 优化(本 milestone):后端看查询/IO/错误处理/复用;前端看渲染/包体/状态/复用/可访问性
6. 验证:测试 + 构建(原则 5)
7. 反思 + 建 ticket(B 轨):当场能改的走第 5 步;值得另起一轮的 create_ticket(技术类自消化 / 产品类标 needs-human)
8. commit-sync:一致性检查 → 更新 spec/进度 → 单 milestone 单 commit/push
9. 检查熔断条件;命中则停并输出摘要
10. 回到 2
```

> **两轨模型**:A 轨(PRD 路线图)是消费主干、默认最高优先级;B 轨(自省 backlog)是 loop 自己生产的工作队列——
> 每 milestone 后反思建 ticket,Room 收尾时**限额 drain**(技术类最多 N 条,带预算上限防"优化无底洞";
> 产品类停着等人批)。详见 [`loop-procedure.md`](./loop-procedure.md) 的「B 轨」一节。
