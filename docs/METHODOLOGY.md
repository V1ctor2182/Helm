# Helm 自动化开发方法论

> 适用对象:用 `/loop` + feature-room skills + VibeHub 驱动的(半)自动开发。
> 这份文档同时是**给人看的完整版**;其中的"决策闸门"与"单/双向门清单"会被精简后写入 VibeHub 约束(`add_constraint`),好让每一轮 loop 通过 `get_team_conventions` 自动读到。
>
> 核心信念:让全自动安全的,不是"原则",而是 **错了也便宜(可回滚)、错了也看得见(留痕)、撞墙会停(熔断)**。原则只决定"每一步怎么想"。
>
> ⚠️ 下面都是**默认倾向,不是死规矩**。拿不准时从宽,优先把活往前推,别为了走流程牺牲进度。阈值/清单都是参考,可以凭判断灵活处理。

---

## 一、六条原则(决策链,按顺序执行)

遇到任何 OQ(Open Question)/ 不确定点,按以下顺序判断:

### 1. 可逆性闸门(先判该不该自己答)
- **单向门(one-way door)** → 默认先别自己答,建 issue + `add_question`,在代码里留显式 TODO,**别写隐式假设了某个答案的代码**。(如果依据很足、答案几乎唯一,也可以自己定,只是要记得 record。)
- **双向门(two-way door)** → 可以自己决。
- 判据:改起来贵不贵?别的模块会不会建在它上面?已经 commit/push 出去后难不难撤?三者任一为"是"即视为单向门。

### 2. 有据可依(决策必须有来源)
- 决策只能来自:PRD(`docs/prd/`)/ VibeHub spec / 已有代码 / 已记录的约束。
- **找不到依据 ≠ 猜**。没有依据的问题,本质就是单向门,走 defer。
- 禁止用"业界常见做法"替代项目内的明确依据来定单向门级决策。

### 3. 取舍排序(仅用于"已判定可自己答"的问题)
- **安全 > 正确/完整 > 速度**。
- 同一问题有"安全但费事"和"省事但有风险"两种解法时,选前者。
- 这条是兜底,不是入口——入口永远是第 1、2 条。

### 4. 留痕(每个自动决策都要记录)
- 自己答掉的决策 → `record_decision`,写清**理由 + 依据来源**。
- 新增的硬性边界 → `add_constraint`。
- 不留痕的后果:下一轮 loop 会重问、甚至答得不一致 → 决策漂移。

### 5. 验证后才提交(commit-sync 前的硬门)
- 提交前必须过:**测试 + 构建/类型检查**(项目早期至少要能 import / 启动成功)。
- 没有验证手段的代码 = 不可全自动。补齐验证门是上"真·无人值守"的前置条件。
- 验证失败 → 不提交,进入熔断判断(见第三节)。

### 6. 有限爆炸半径(约束影响范围)
- **一个 Room = 一条分支**(如 `feat/platform-shell`)。绝不在 `main` 上自动开发。
- 在该分支上把这个 Room 的 milestone 逐个做完;**单 milestone 单 commit**,便于整段回滚。
- Room 收尾 → 产出复盘(`room-status`)→ 开 PR 合回 `main`,人确认后再合。
- **下一个 Room 从最新的 `main` 切新分支**,这样它能拿到前序 Room 已合入的成果(处理 Room 间依赖,如大家都依赖 `platform-shell` 地基)。
- 因此 **Room 之间是串行**的:做完一个、合一个、再切下一个,避免并行分支互相依赖时冲突。

## 二、Helm 的单向门 / 双向门清单

> 这是把原则落到本项目的清单,当参考用。新出现、拿不准的决策,顺手 `add_question` 记一句问我,但别因此卡住——能继续的部分继续做。

### 单向门 —— defer / 问人,不准 loop 自己定

- **数据模型 / 本地存储 schema**:任何会导致后续数据迁移的结构。
- **对外契约**:
  - 暴露给 Claude Code 的 MCP 工具签名与语义(`agent-orchestration`)。
  - 跨 Room 契约:驾驶舱如何观测 agent 产出(`cockpit`)、memory/RAG 的 MCP 表面(`memory-rag-skills`)。
- **安全边界**:
  - 凭据/密钥的存储位置与加密方式。
  - 哪些数据会发送给外部 provider(`chat-multimodel` / `deep-research`)。
  - 终端 agent(Claude Code)的沙箱与权限边界。
- **核心选型**:后端框架、桌面外壳(pywebview vs Electron)、关键第三方依赖、打包方案(py2app / PyInstaller)的最终确定。
- **跨 Room 共享的基础抽象**:多个 feature 会依赖的公共模块接口。

### 双向门 —— loop 自己决 + `record_decision`

- Room **内部**的命名、文件/目录组织。
- 内部 helper / 工具函数的实现方式。
- 测试的组织与命名。
- 非对外 UI 组件的内部结构与状态管理细节。
- 日志格式、报错文案、注释。
- 内部重构(不改对外行为)。

## 三、熔断条件(任一触发 → 停止 loop,等人)

全自动的安全性主要来自"会停下来"。下面阈值是**参考值,不用死扣**;命中其一,大致就该停下来产出状态摘要、等我介入:

1. **测试/构建连续失败 2 次**(同一 milestone)。
2. **单向门撞墙**:一个 milestone 里出现 ≥1 个无法自动决策的单向门 OQ,且它阻塞了后续工作。
3. **打转**:同一文件在一轮内被反复修改 ≥3 次仍未通过验证。
4. **defer 堆积**:单个 Room 累计未解决 issue/question ≥ 5 条。
5. **越界**:改动触及单向门清单中的文件/契约,而当前 milestone 并未授权。
6. **依据缺失**:连续 ≥2 个 OQ 因"找不到依据"而无法推进。
7. **范围漂移**:实际改动超出当前 milestone 声明的范围(commit-sync 一致性检查不通过)。

熔断时必须输出:停在哪个 Room/milestone、触发了哪条、已 defer 的 issue 列表、建议的下一步。

## 四、自动化成熟度分档

| 档位 | 前置条件 | 人的角色 |
|------|----------|----------|
| **监督式自动** | 六条原则 + 单/双向门清单 + 熔断已配置 | 每个 Room 跑完看 `room-status` 复盘 |
| **真·无人值守** | 额外需要:验证门(测试/CI)就绪、熔断真的会触发、dev 分支 + 单 milestone 单 commit | 仅按熔断/复盘节点介入 |

**当前状态**:仓库尚无代码与测试 → 暂不具备"真·无人值守"前置条件。**先用监督式自动跑通 `platform-shell`,并在此过程中把测试/验证门建起来**,再考虑切换。

## 五、Milestone 拆分原则(plan-milestones 阶段适用)

> 目标:**能在当前 milestone 做完的,就当场做完,不要顺手丢一个 "留到 phase 5" 的 TODO。** 拖延式拆分会让代码长期处于半成品状态、stub 满天飞,最后没人记得回去补。

1. **就近做完(close it out)**:一个功能如果在当前 milestone 的能力范围内就能完整实现,就完整实现掉,不要拆成"先搭壳、后面再填"。
2. **垂直切片优于水平分层**:每个 milestone 交付一个**端到端可验证**的薄切片(从数据到 UI 打通一条),而不是"这个 milestone 只搭后端、UI 等以后"。完成后这块应当是真能跑、能验证的,而非一堆待接线的桩。
3. **禁止隐性拖延**:不允许用 "TODO: phase N 再做" 来回避当前就能完成的工作。真正需要推迟的,只有两类——
   - **单向门**(需要人拍板,见第二节)→ 走 defer + 建 issue,这是合法推迟。
   - **明确超出本 Room 范围**的工作 → 记成另一个 Room/feature 的 issue。
   除此之外,"能做但懒得现在做"不是合法的推迟理由。
4. **拆分粒度**:milestone 要小到"单 milestone 单 commit、可整段回滚"(原则 6),但不要小到把一个内聚的功能硬切成"搭壳 / 填肉"两半。**以"完成一个可验证的能力"为单位,而不是以"碰一层代码"为单位。**
5. **完成定义(DoD)**:一个 milestone 算完成 = 该切片功能可用 + 通过验证门(原则 5)+ 没有为了赶进度留下的隐式假设/半接线 stub。

冲突时的优先级:**完整地做完一个小切片 > 铺开很多个半成品**。

## 六、单轮 loop 的标准动作

```
1. pull context:get_feature_context / query_why / get_team_conventions
2. 取下一个 milestone(无则停)
3. 开发;遇 OQ → 走第一节"六条原则"决策链
   - 双向门:决 + record_decision
   - 单向门:add_question + 建 issue + 代码留显式 TODO,不写隐式假设
4. 验证:测试 + 构建(原则 5)
5. commit-sync:一致性检查 → 更新 spec/进度 → 单 milestone 单 commit/push
6. 检查熔断条件;命中则停并输出摘要
7. 回到 2
```
